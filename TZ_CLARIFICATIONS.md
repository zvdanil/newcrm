## КРИТИЧЕСКИЕ РЕШЕНИЯ (Архитектура БД)

**П.1 — Owner "удаление" записей**
Физический DELETE запрещён. Owner делает Soft Delete: запись получает is_deleted = true,
система автоматически генерирует сторнирующую транзакцию для выравнивания баланса.
Реляционная целостность сохраняется полностью.
**How to apply:** Все финансовые таблицы требуют колонки is_deleted + deleted_at + deleted_by.
При soft-delete триггер создаёт REVERSAL транзакцию текущей датой.

**П.2 — Тип возврата за питание → Вариант B (окончательно)**
Полный отказ от refund_food_deferred и скрытых счётчиков.
Единственный тип: REFUND (мгновенный) — сразу увеличивает баланс ребёнка.
ACCRUAL 1-го числа всегда создаётся на ПОЛНУЮ сумму Base_Fee (без вычетов).
Итоговый баланс = Оплаты + Возвраты - Начисления.
Сумма к оплате очевидна из баланса — не нужна отдельная формула.
**How to apply:** В Billing Run убрать компонент refund_food_deferred. ACCRUAL = Base_Fee всегда.

**П.3 — advance_balance: структура**
Аванс хранится РАЗДЕЛЬНО по финансовому счёту.
Структура таблицы балансов: (child_id, account_id, balance).
Единого "общего аванса" ребёнка нет — только per-(child, account).
**How to apply:** Таблица child_balances: PK = (child_id, account_id).

## АЛГОРИТМИЧЕСКИЕ РЕШЕНИЯ

**П.4 — Waterfall: приоритет погашения**
1. FIFO — сначала самый старый долг (по transaction_date ASC).
2. При одинаковой дате — приоритет самому крупному долгу (amount DESC).
**How to apply:** ORDER BY transaction_date ASC, amount DESC в запросе открытых долгов.

**П.5 — Отрицательное начисление → не существует**
ACCRUAL всегда = полная Base_Fee. Аванс не уменьшает сумму транзакции ACCRUAL.
Аванс перекрывает долг только на уровне расчёта баланса (формула).
Если аванс > долга — баланс положительный (аванс), новое начисление это изменит.
**How to apply:** Guard в Billing Run не нужен. Всегда создавать ACCRUAL = Base_Fee.

**П.6 — Quick Enrollment: pro-rata при записи mid-month**
При создании enrollment через Quick Enrollment:
- Система рассчитывает пропорциональную стоимость за оставшиеся дни месяца.
- Формула: (Base_Fee / рабочих_дней_в_месяце) × дней_до_конца_месяца.
- Немедленно создаётся транзакция ACCRUAL на эту сумму.
- С 1-го числа следующего месяца — стандартный Billing Run на полный Base_Fee.
**How to apply:** После INSERT в enrollments → триггер pro-rata ACCRUAL если mid-month.

**П.7 — Заморозка: видимость педагогу**
Ребёнок отображается в журнале с визуальным индикатором заморозки (иконка/серый фон).
Ячейки журнала для замороженного ребёнка: disabled (нельзя нажать).
Финансовых транзакций не генерируется.
**How to apply:** В запросе журнала включать замороженных детей, передавать флаг is_frozen.
На фронте: row с is_frozen → все cells disabled + визуальный маркер.

**П.8 — Синхронные активности: каскад отсутствия**
Отметка "Уважительная причина" в главной активности → автоматически транслируется
во все связанные активности с тем же статусом.
Каждая связанная активность применяет СВОИ правила возврата независимо.
Пример: Садик (жёсткий абонемент) → возврата нет. Питание (связанная) → REFUND за еду.
**How to apply:** После INSERT в attendance_log (parent activity) → loop по linked_activities
→ INSERT в attendance_log каждой → каждая сама вычисляет свой refund по своим настройкам.

## РОЛЕВЫЕ И ОПЕРАЦИОННЫЕ РЕШЕНИЯ

**П.9 — Менеджер и финансовые транзакции**
Менеджер регистрирует СОБЫТИЯ (факт прихода ребёнка), а не финансы.
Финансовые транзакции генерирует Billing Engine на сервере по тарифу Admin/Owner.
Менеджер НЕ может: редактировать тарифы, списывать долги вручную, переводить деньги.
Это архитектурно безопасно — UI менеджера не имеет доступа к финансовым эндпоинтам.
**How to apply:** RBAC: роль Manager не имеет права на /api/transactions (POST/PUT/DELETE).
Отметки в журнале → /api/attendance → Billing Engine создаёт транзакции сам.

**П.10 — Семейный платёж (Waterfall по нескольким детям)**
Поддерживается семейный Waterfall. При оплате в семейном аккаунте:
- Система собирает все долги ВСЕХ детей семьи по указанному счёту (account_id).
- Применяет общее правило: FIFO → при равной дате — наибольшая сумма.
- Опционально: Admin может вручную выбрать, какие именно долги гасить (Manual Match).
**How to apply:** Waterfall-запрос: WHERE child_id IN (дети семьи) AND account_id = X.

**П.11 — Кросс-счётные оплаты (ФОП 1 / ФОП 2)**
- Долг ребёнка по услуге ФОП 1 → закрывается (ребёнок ничего не должен).
- Реальные деньги зачисляются на ФОП 2 (баланс ФОП 2 растёт).
- Баланс физических денег на ФОП 1 НЕ увеличивается искусственно.
- В карточке счёта ФОП 1 отображается информационный "Межсчётный дисбаланс":
  "ФОП 2 должен перевести на ФОП 1 сумму X грн". Виден Owner и Admin.
**How to apply:** Отдельная таблица inter_account_imbalances (from_account, to_account, amount).
Отображается как информационный блок в карточках счетов.

**П.12 — Пагинация: архитектура отображения**
Бэкенд: LIMIT 500 (нормально для SQL, снижает кол-во запросов).
Фронтенд: виртуализация списков (react-window или @tanstack/virtual) — рендерится
только видимая область, 500 строк не нагружают DOM.
Альтернатива: постраничное отображение по 50 записей с серверной пагинацией.
**How to apply:** Обсудить с фронтендом — виртуализация предпочтительнее для UX.

**П.13 — "Ожидает оплаты" ≠ "Прогноз"**
Прогноз (отклонён) = абстрактные плановые суммы без факта события.
Ожидает оплаты = Кредиторская задолженность (Accounts Payable):
услуга/работа уже выполнена, обязательство уже возникло, деньги физически не списаны.
Это строгий бухгалтерский учёт, а не бюджетирование.
**How to apply:** Статус "pending_payment" у расходных транзакций = AP. В PnL учитывается
в колонке "Начисленные расходы" с момента создания (до фактической выплаты).

**П.14 — Жёсткий абонемент + уважительная причина**
Жёсткий абонемент блокирует возврат ТОЛЬКО за основную услугу (обучение/присмотр).
Питание — отдельная синхронная связанная активность со своими правилами.
При уважительной причине: Садик (жёсткий) → возврата нет. Питание → REFUND_IMMEDIATE.
**How to apply:** Логика возврата проверяется в каждой активности отдельно по её refund_config,
не наследуется от родительской активности.

## РЕШЕНИЯ — СЕССИЯ 2026-05-07

**П.15 — Индивидуальные тарифы для детей (child_prices)**
Отдельная карточка "Индивидуальные тарифы" не нужна — UI перегружается.
Управление тарифом встроено прямо в строку подписки (enrollment row) как инлайн-форма.
Хранятся по SCD Type 2: valid_from / valid_to. Активный тариф = valid_to IS NULL.
Удаление = установка valid_to (не физический DELETE). Будущая дата = планирование перехода.
**How to apply:** DELETE /prices/:id?valid_to=YYYY-MM-DD → устанавливает valid_to.
Если valid_to в будущем — тариф ещё действует, но запланирован сброс.
В Billing Run и при per_lesson accrual — child_prices проверяется первым по иерархии.

**П.16 — Смарт-тариф (tariff_type='smart'): архитектура**
Третий тип тарифа после 'monthly' и 'per_lesson'. Гибкая логика для нетипичных ситуаций.
Конфигурируется в карточке активности через SmartTariffConfigBlock.

ACCRUAL = B (полная база) создаётся 1-го числа через отдельный `runSmartAccruals` в cron.
REFUND = benefit пересчитывается в реальном времени после каждого изменения отметки.

**Логика 1 (threshold_fee):** если excused_absences ≥ l1_threshold_absences → benefit_L1 = B - l1_threshold_fee.
**Логика 2 (capped_refund):** benefit_L2 = min(absences, l2_max_refunds) × l2_refund_per_absence.
**Комбинация:** benefit = max(benefit_L1, benefit_L2). Обе логики могут быть активны одновременно.
**Ноль пропусков = ноль пользы.** Рефанд не создаётся.

Идентификация REFUND: metadata_json->>'source' = 'smart_benefit'. Один такой REFUND
на enrollment+billing_month. При пересчёте: старый soft-deleted, создаётся новый.
**How to apply:** На каждое изменение attendance → `recalcSmartBenefit(enrollmentId, billingMonth)`.
Таблица `smart_tariff_configs` — один конфиг на активность (activity_id PRIMARY KEY).

**П.17 — Детализация Ledger по месяцам: отображение**
Показывать итоги per-activity (сгруппировано), а не отдельные строки каждой транзакции.
Навигация по месяцам; суммы считаются бэкендом по всему диапазону.
**How to apply:** Фронтенд группирует транзакции по activity_name внутри выбранного месяца,
показывает нетто-сумму. Кликабельность строк — на усмотрение (расширение при необходимости).

## РЕШЕНИЯ — СЕССИЯ 2026-05-08 → 2026-05-11

**П.18 — SCD Type 2: граница valid_to — exclusive end**
Поле `valid_to` всегда трактуется как исключительная граница (`<`, не `<=`).
Запрос "тариф действует на дату D" = `valid_from <= D AND (valid_to IS NULL OR valid_to > D)`.
**Why:** При закрытии тарифа через DELETE default `valid_to = firstOfCurrentMonth`.
Если бы граница была `>=`, тариф продолжал бы действовать в день своего закрытия.
**How to apply:** Во всех функциях: `getChildIndividualTariff`, `getEffectivePrice`, `triggerRefund`,
`triggerPerLessonAccrual` — использовать `>` для valid_to. Никогда `>=`.

**П.19 — Дата закрытия тарифа по умолчанию = первое число текущего месяца**
При закрытии индивидуального тарифа / цены через UI дефолт = `firstOfCurrentMonth()`.
**Why:** Закрытие тарифа означает "за текущий месяц тариф уже не должен применяться".
Если бы дефолт был `today`, тариф-граница попадала бы в середину месяца — пересчёт был бы частичным.
**How to apply:** В `children.ts` DELETE-эндпоинты: `valid_to ?? firstOfCurrentMonth`.
В фронтенде `ChildCardPage.tsx`: инициализировать форму закрытия через `firstOfMonth()`.

**П.20 — Отмена ACCRUAL: каскад на отметки и возвраты**
При отмене ACCRUAL через `POST /transactions/:id/cancel`:
- **Monthly / smart**: soft-delete всех REFUND за enrollment+billing_month + физическое удаление `absent_excused` из `attendance_logs` за тот же месяц. Каскадируется на все linked-активности.
- **Per-lesson**: физическое удаление `present/special` отметки из `attendance_logs` за дату ACCRUAL.
**Why:** Отмена начисления означает "этот расчётный период аннулирован". Отметки, порождавшие
финансовые следствия этого периода, теряют смысл и должны быть удалены вместе с транзакциями.
**How to apply:** В `transactions.ts` — каскад реализован после soft-delete ACCRUAL.
Linked-активности определяются через таблицу `linked_activities`.

**П.21 — REFUND дублирование при upsert отметки: защита**
При POST /attendance (upsert) статус `absent_excused` НЕ создаёт дублирующий REFUND,
если активный REFUND для этого enrollment+date уже существует.
**Why:** POST-обработчик работает как upsert (onConflict doUpdate). При повторном нажатии
той же отметки `triggerRefund` вызывался снова → второй REFUND.
**How to apply:** Перед `triggerRefund` проверять наличие активного REFUND для enrollment+date.
Если есть — пропустить. То же для каждой linked-активности.
