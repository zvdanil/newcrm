## 1. ЧТО СТРОИМ

**КіндерCRM / IRIS** — ERP-система для детского образовательного центра/садика.

Главная особенность: **посещаемость напрямую генерирует финансовые транзакции**. Галочка педагога в журнале = бухгалтерская проводка.

### Три слоя системы
1. **Клиентский блок** — семья → дети → подписки на активности → раздельные балансы
2. **Биллинговое ядро (Billing Engine)** — асинхронные триггеры от журналов + Billing Run 1-го числа
3. **Финансовый учёт** — расходы (метод начислений), зарплаты, дивиденды, PnL

## 3. АРХИТЕКТУРНЫЕ ПРИНЦИПЫ (НЕЛЬЗЯ НАРУШАТЬ)

### 3.1 Иммутабельность финансовой истории
- Физический `DELETE` финансовых записей **запрещён для всех ролей**.
- Owner использует **Soft Delete**: запись получает `is_deleted = true`, система **автоматически** создаёт сторнирующую транзакцию для выравнивания баланса.
- Любая корректировка = новая транзакция (ADJUSTMENT / REVERSAL).

### 3.2 Версионность (SCD Type 2)
Все сущности со стоимостью (тарифы, скидки, ставки педагогов) хранятся с временными интервалами:
- `valid_from` — дата начала действия
- `valid_to` — дата окончания (`NULL` = актуальная запись)

### 3.3 Ретроспективный пересчёт
При изменении цены "задним числом" система **не перезаписывает** старые транзакции — рассчитывает дельту и создаёт корректировочную запись (ADJUSTMENT) текущей датой.

## 4. СХЕМА ДАННЫХ (КЛЮЧЕВЫЕ РЕШЕНИЯ)

### 4.1 Балансы
```sql
-- Баланс хранится РАЗДЕЛЬНО по финансовому счёту
-- Единого "общего аванса" ребёнка НЕТ
child_balances (
  child_id    UUID,
  account_id  UUID,   -- ФОП 1, ФОП 2, etc.
  balance     NUMERIC -- > 0 = аванс, < 0 = долг
  PRIMARY KEY (child_id, account_id)
)
```

### 4.2 Формула баланса
```
Баланс = Оплаты + Возвраты (REFUND) - Начисления (ACCRUAL)
Баланс > 0 → Аванс
Баланс < 0 → Задолженность
```

### 4.3 Типы транзакций
| Тип | Описание |
|---|---|
| `ACCRUAL` | Начисление долга клиенту (всегда = `Base_Fee`, без вычетов) |
| `PAYMENT` | Оплата от клиента |
| `REFUND` | Мгновенный возврат (увеличивает баланс) |
| `ADJUSTMENT` | Корректировочная запись при ретро-пересчёте |
| `REVERSAL` | Сторно при soft-delete Owner-ом |
| `SALARY_ACCRUAL` | Начисление зарплаты педагогу |
| `SALARY_PAYMENT` | Выплата зарплаты |
| `EXPENSE_ACCRUAL` | Начисленный расход (Accounts Payable) |
| `EXPENSE_PAYMENT` | Фактическая выплата расхода |

### 4.4 Обязательные индексы
```sql
CREATE INDEX ON transactions (child_id);
CREATE INDEX ON transactions (staff_id);
CREATE INDEX ON transactions (account_id);
CREATE INDEX ON transactions (transaction_date);
CREATE INDEX ON transactions (category_id);
```

## 5. БИЛЛИНГОВОЕ ЯДРО (BILLING ENGINE)

### 5.1 Возврат за питание / отсутствие
**Только один тип — мгновенный `REFUND`**. Отложенных возвратов (`refund_food_deferred`) не существует.

При отметке "отсутствовал по уважительной причине":
1. Создаётся транзакция `REFUND` → баланс ребёнка сразу растёт.
2. Аванс накапливается и уменьшит следующий `ACCRUAL` через формулу баланса.

### 5.2 Billing Run (1-е число каждого месяца)
```
ACCRUAL = Base_Fee  ← ВСЕГДА полная сумма, аванс НЕ вычитается из транзакции
```
Итоговая сумма к оплате видна родителю через баланс: `Base_Fee - advance_balance`.

**Важно:** Отрицательный `ACCRUAL` невозможен. Если аванс > Base_Fee — аванс остаётся, новое начисление создаётся на `Base_Fee`, баланс становится ещё более положительным.

### 5.3 Quick Enrollment (запись mid-month)
При записи ребёнка не 1-го числа:
1. Немедленно создаётся `ACCRUAL` с **pro-rata** расчётом:
   ```
   pro_rata = (Base_Fee / рабочих_дней_в_месяце) × дней_до_конца_месяца
   ```
2. С 1-го числа следующего месяца — стандартный Billing Run.

### 5.4 Waterfall (распределение платежа)
Приоритет погашения долгов:
1. **FIFO** — сначала самый старый долг (`transaction_date ASC`)
2. При одинаковой дате — самый крупный долг (`amount DESC`)

```sql
SELECT * FROM open_debts
WHERE child_id = ? AND account_id = ?
ORDER BY transaction_date ASC, amount DESC
```

### 5.5 Семейный Waterfall
При оплате в семейном аккаунте система собирает долги **всех детей семьи** по указанному `account_id` и применяет то же правило (FIFO → largest). Admin может выбрать Manual Match для конкретных долгов.

### 5.6 Кросс-счётные оплаты (ФОП 1 / ФОП 2)
Деньги пришли на ФОП 2, но услуга привязана к ФОП 1:
- Долг ребёнка по ФОП 1 → закрывается (клиент ничего не должен)
- Физические деньги → зачисляются на ФОП 2
- Баланс ФОП 1 физически **не увеличивается**
- Создаётся запись в `inter_account_imbalances(from_account, to_account, amount)` — информационный "Межсчётный дисбаланс" видный Owner и Admin

### 5.7 Смарт-тариф (`tariff_type = 'smart'`)
Третий тип тарифа. Конфигурируется в карточке активности. Работает по двум независимым логикам:

**Логика 1 (threshold_fee):** если уважительных пропусков ≥ `l1_threshold_absences` → клиент платит `l1_threshold_fee` вместо базы. `benefit_L1 = B - l1_threshold_fee`.

**Логика 2 (capped_refund):** первые `l2_max_refunds` уважительных пропусков получают возврат `l2_refund_per_absence` каждый. `benefit_L2 = min(absences, l2_max_refunds) × l2_refund_per_absence`.

**Правило комбинации:** `benefit = max(benefit_L1, benefit_L2)`. Ноль пропусков → ноль пользы.

**Механизм:**
1. `ACCRUAL = B` создаётся 1-го числа через `runSmartAccruals` (cron, отдельно от Billing Run).
2. После каждого изменения отметки в журнале → `recalcSmartBenefit(enrollmentId, billingMonth)`.
3. Создаётся/обновляется единственный `REFUND` с `metadata_json->>'source' = 'smart_benefit'` для данного enrollment+billing_month.
4. При пересчёте: старый REFUND soft-deleted, создаётся новый с актуальной суммой.

**Таблица конфигурации:** `smart_tariff_configs` (activity_id PRIMARY KEY, base_lessons, l1_threshold_absences, l1_threshold_fee, l2_max_refunds, l2_refund_per_absence).

## 6. ИЕРАРХИЯ ЦЕНООБРАЗОВАНИЯ

Применяется в строгом порядке (1 = высший приоритет):
1. Спец-тариф (ручной ввод в ячейке журнала)
2. Индивидуальная цена ребёнка на конкретную активность (`child_prices`)
3. Скидка на активность для ребёнка (`child_prices.discount_pct`)
4. Глобальная скидка ребёнка (`child_global_discounts`)
5. Базовая цена активности (`tariffs.base_fee`)

**Индивидуальные тарифы** хранятся по SCD Type 2 (valid_from / valid_to). Сброс тарифа = установка valid_to (поддерживает планирование будущим числом). Управление встроено в строку подписки в карточке ребёнка (не отдельный блок).

## 7. ЖУРНАЛЫ И АКТИВНОСТИ

### 7.1 Заморозка подписки
- Ребёнок **остаётся** в журнале педагога
- Строка визуально помечена (иконка / серый фон)
- Ячейки отметок **заблокированы** (`disabled`) — транзакции не генерируются

### 7.2 Синхронные (связанные) активности — каскад отсутствия
Отметка в главной активности → автоматически транслируется во все связанные.
**Каждая активность применяет СВОИ правила возврата независимо.**

Пример: Садик (жёсткий абонемент) + Питание (связанная):
- Уважительная причина в Садике → отметка ставится в Питание
- Садик: возврата **нет** (жёсткий абонемент блокирует возврат за основную услугу)
- Питание: `REFUND` за еду (своя конфигурация)

### 7.3 Жёсткий абонемент
- Блокирует возврат **только за основную услугу**
- Питание как связанная активность — **не блокируется**, возврат идёт по её конфигурации

## 8. РОЛИ И ПРАВА ДОСТУПА (RBAC)

| Роль | Ключевые права |
|---|---|
| **Owner** | Полный доступ, soft-delete, дивиденды, PnL, межсчётные операции |
| **Admin** | Дети, журналы, оплаты, тарифы. Не видит чужие ЗП, дивиденды, PnL |
| **Manager** | Отметки в журналах, просмотр. НЕ управляет финансами напрямую |
| **Accountant** | Счета, расходы, ведомости ЗП. Дети — только в части оплат |
| **Teacher** | Своё расписание, отметки в своих журналах, своя начисленная ЗП |
| **Parent** | Только свой ребёнок: баланс, история оплат, посещения |

**Важно про Manager:** Manager регистрирует события (факт прихода), не финансы.
Транзакцию генерирует Billing Engine на сервере по тарифу Admin/Owner.
UI менеджера **не имеет доступа** к `/api/transactions` (POST/PUT/DELETE).

## 9. РАСХОДЫ (МЕТОД НАЧИСЛЕНИЙ)

Два статуса расходной транзакции:
- **`EXPENSE_ACCRUAL`** (Accounts Payable) — услуга оказана, обязательство возникло, деньги не списаны. **Это НЕ прогноз.** Входит в PnL как "Начисленный расход".
- **`EXPENSE_PAYMENT`** — фактическое списание со счёта при нажатии "Оплатить"

Прогнозные/бюджетные флаги — **отсутствуют в системе**.

---

## 10. PnL ОТЧЁТ

Генерируется **только по кнопке** (не при открытии страницы). Колонки по месяцам:

| Колонка | Источник |
|---|---|
| Ожидаемый доход | Сумма всех ACCRUAL клиентам |
| Начисленный расход | EXPENSE_ACCRUAL + SALARY_ACCRUAL |
| Реальные доходы | Фактически поступившие PAYMENT |
| Оборот расходов | EXPENSE_PAYMENT + SALARY_PAYMENT |
| Оборот без дивидендов | То же, исключая транзакции с флагом `is_dividend` |
| Баланс без дивидендов | Реальные доходы − Оборот без дивидендов |
| Остаток на счетах | Фактическая ликвидность всех счетов (накопительно) |

## 11. ПРОИЗВОДИТЕЛЬНОСТЬ

- **Бэкенд:** `LIMIT 500` на все списочные запросы
- **Фронтенд:** Виртуализация (`react-window` или `@tanstack/virtual`) — рендерится только видимая область. DOM никогда не содержит 500 строк одновременно.
- **Итоги:** Суммы считаются бэкендом по всему отфильтрованному массиву в БД, а не по 500 видимым записям.

---

## 12. СОГЛАШЕНИЯ ПО КОДУ

- Все финансовые расчёты — **только на бэкенде**. Фронтенд отображает результаты.
- Финансовые суммы хранятся как `NUMERIC(15,2)` (не `FLOAT`).
- Все транзакции имеют `metadata_json` — слепок тарифа/ставки на момент создания.
- Временные метки — `TIMESTAMPTZ` (с учётом часового пояса).
- Мягкое удаление: каждая финансовая таблица имеет `is_deleted BOOLEAN DEFAULT false`, `deleted_at TIMESTAMPTZ`, `deleted_by UUID`.

## 13. ОТКРЫТЫЕ ВОПРОСЫ / УТОЧНИТЬ ПОЗЖЕ

- Точная формула рабочих дней для pro-rata (календарные или только будни?)
- Порядок детей в семейном Waterfall при нескольких детях (старший первый или по FIFO долгов?)
- Как именно отображается "Межсчётный дисбаланс" в карточке счёта (отдельный виджет или строка?)
- Максимальный срок заморозки подписки
